# 涨停板概念分组 — 执行任务说明书

## 任务目标

对每日涨跌停数据，完成概念分级归类，输出带层级结构（一级→二级→三级）的 JSON。

---

## 一、目录结构

```
题材分析/                         # 项目根目录
├── concepts_all.json             # [核心参考] 所有个股的概念分类数据（单文件）
├── py/                          # 脚本
│   ├── run_中间.py              # Step 1: 噪音过滤 + concepts_all.json 匹配 → 产出 real_concepts
│   ├── split_groups.py          # Step 2: 将股票分为 5 组，供子 LLM 并行分析
│   └── validate_stocks.py       # Step 6: 验证原始数据与最终输出的个股一致性
│
├── 题材分类分析.md               # 本说明书
├── 热门概念.json                 # 热点题材配置（可选）
├── limit_up/                    # 原始数据（涨停板JSON）
├── limit_up_中间/               # 中间数据（分组文件、子 LLM 结果）
│   └── {date}_涨停概念分组_中间.json  # ⚠️ 唯一保留的中间文件，其余过程文件在工作流完成后清理
└── limit_up_分析/               # 输出数据（最终结果）
```

### 各步骤速查

| 步骤       | 谁做                 | 做什么                                                | 产出                                                              |
| ---------- | -------------------- | ----------------------------------------------------- | ----------------------------------------------------------------- |
| **Step 1** | 🖥️ 脚本              | `python py/run_中间.py {date}`                        | `limit_up_中间/{date}_涨停概念分组_中间.json`（含 real_concepts、market_concept、reason_info） |
| **Step 2** | 🖥️ 脚本 + 🧠 5×子LLM | `python py/split_groups.py {date}` → 5个子LLM各自分析 | `limit_up_中间/{date}_group_{1..5}_result.json`                   |
| **Step 3** | 🧠 LLM               | 汇总5组结果 + 交叉验证，直接产出最终文件              | `limit_up_分析/{date}.json`（不再产出 \_merged.json）             |
| **Step 5** | 🖥️ 脚本              | `node py/gen_豆包模板.js {date}`                      | `豆包调研模板/{date}_豆包调研模板.md`                             |
| **Step 6** | 🖥️ 脚本              | `python py/validate_stocks.py {date}`                  | 验证通过 ✅ / 失败 ❌ 需修复                                     |

### Step 2 子 LLM 并行分析说明

**split_groups.py** 将股票按涨停时间均匀分为 5 组，每组文件嵌入完整的分析指令和热门概念信息。5 个子 LLM 各自分析一组，互不干扰。

每个子 LLM 读取分组文件后，对每只股票分析以下字段：

| 字段              | 来源                         | 说明                               |
| ----------------- | ---------------------------- | ---------------------------------- |
| `real_concepts`   | `concepts_all.json` 自动匹配 | 参考数据中已有的概念分类，直接可用 |
| `primary_concept` | 同花顺信号标签               | 市场给它的主要标签（频次最高）     |
| `concepts`        | 同花顺信号标签               | 市场给它的所有标签                 |
| `market_concept`  | 同花顺涨停简图实时分类       | 市场给它的涨停概念（如"算力/半导体产业链"），可用于补充/确认 real_concepts |
| `reason_info`     | 板块涨停排行 API             | 涨停原因详情，用于提取时效性数据写入 note |

**子 LLM 的判断逻辑（已嵌入分组文件）：**

`concepts_all.json` 只是历史整理的静态参考数据，**不是绝对正确的标准**。所有股票都需要 LLM 综合以下因素判断：

1. **热门概念** — 该股票是否匹配 `热门概念.json` 中的热点题材？热门概念优先
2. **概念数量（板块效应）** — 同一天涨停股中，同一概念出现的数量越多，板块效应越强，越应该采用
3. **市场信号** — `primary_concept`、`concepts`、`market_concept` 是市场给它的实时标签，有重要参考价值
4. **concepts_all.json** — 参考数据中已有的分类，判断是否仍然有效（可能滞后）
5. **业务实质** — 股票的实际业务是否真的与分类匹配

**判断流程：** 综合以上因素，选择最合适的分类。concepts_all.json 只是参考之一，不是最终答案。一只股票可能有多个概念 → 在 memberships 数组中添加多条。

### Step 3 LLM 汇总 + 交叉验证说明

读取 5 份 `_result.json`，合并并交叉验证，直接产出最终文件：

1. **合并** — 把 5 组的所有记录合并到一份列表
2. **检查重复** — 同一只股票如果有重复概念记录，合并或去重
3. **同概念一致性** — 同一概念下的股票，归类是否一致？（比如"PCB与载板"的股票应该都在同一组概念下）
4. **明显错误** — 是否有明显错误的概念判断（比如把消费股归类到 AI 硬件）
5. **跨组一致性** — 5 个子 LLM 的分析标准是否一致？需要统一调整
6. **产出** — 直接产出 `limit_up_分析/{date}.json`（不再产出 `_merged.json` 中间文件）

### 数据格式

```json

        "strength": 5,

      }
    ]
  },
  {
    "symbol": "002129.SZ",
    "name": "TCL中环",
    "memberships": [
      {
        "sub_concept": ["半导体", "半导体材料", "硅片/衬底"],
        "strength": 4,
        "note": "光伏硅片双寡头之一..."
      },
      {
        "sub_concept": ["光伏", "硅料与硅片"],
        "strength": 4,
        "note": "光伏硅片双寡头之一..."
      }
    ]
  }

```

### 层级推导规则

`sub_concept` 数组直接就是概念层级路径：

| sub_concept 数组                    | 输出层级                          |
| ----------------------------------- | --------------------------------- |
| `["AI应用", "AI语料"]`              | 二级：AI应用 → AI语料             |
| `["AI硬件", "PCB与载板", "覆铜板"]` | 三级：AI硬件 → PCB与载板 → 覆铜板 |
| `["机器人", "电机"]`                | 二级：机器人 → 电机               |

**规则：**

- `sub_concept` 数组的**最后一个元素**是细分概念
- **前面的元素**是层级路径（一级、二级……）
- 输出时直接使用整个数组，无需再从文件路径推导

### 匹配方式

按 `symbol`（股票代码）精确匹配。一只股票可能出现在多条 memberships 记录中（多概念），此时输出一个条目包含多个 memberships。

## 四、匹配规则（严格按此执行）

### 4.1 概念层级推导规则

参考数据的 `sub_concept` 已经是数组，直接使用：

```
sub_concept 数组                    →  输出数组
─────────────────────────────────────────────────────────
["半导体", "半导体材料", "硅片/衬底"]  →  ["半导体", "半导体材料", "硅片/衬底"]
["AI硬件", "PCB与载板", "覆铜板"]    →  ["AI硬件", "PCB与载板", "覆铜板"]
["机器人", "电机"]                  →  ["机器人", "电机"]
["光伏", "硅料与硅片"]              →  ["光伏", "硅料与硅片"]
```

**规则：** 直接使用 `sub_concept` 数组，无需拼接或推导。

### 4.2 匹配优先级

1. **按 symbol（股票代码）精确匹配** — 在 `concepts_all.json` 中查找
2. 一个股票可能有**多个 memberships**（如 TCL中环 同时在 `半导体/半导体材料` 和 `光伏` 下），此时在同一个条目的 memberships 数组中添加多条
3. 如果在参考数据中找不到 → **进入 AI 补全流程**

### 4.3 参考数据与中间数据冲突时

**concepts_all.json 不是绝对正确的标准。** 它只是历史整理的静态参考数据，可能滞后或过时。LLM 需要综合以下因素判断：

1. **热门概念** — 如果该股票匹配 `热门概念.json` 中的热点题材，优先使用热门概念的层级
2. **概念数量** — 同一天涨停股中，同一概念出现的数量越多，说明板块效应越强，越应该采用
3. **市场信号** — 中间数据的 `primary_concept`、`concepts`、`market_concept` 是市场给它的实时标签，有参考价值
4. **业务实质** — 股票的实际业务是否真的与 concepts_all.json 中的分类匹配

**判断逻辑：** 综合以上因素，选择最合适的分类。concepts_all.json 只是参考之一，不是最终答案。

例如：

```
祥鑫科技:
  - concepts_all.json → ["电池", "结构件"]
  - 中间数据 primary_concept = "液冷服务器"
  - 当天涨停股中有 5 只以上与液冷/散热相关
  → 优先考虑 ["AI硬件", "散热", "液冷散热"]
```

### 4.4 AI 补全规则（参考数据未覆盖的股票）

1. 先用**自己的知识**判断股票的核心题材归属
2. 拿不准时，允许**联网搜索**确认
3. 优先使用同一天其他涨停股票的板块效应来推断（板块效应很重要）
4. 如果多个概念都有依据，允许多条记录
5. **尽量不填 `["未知"]`**，只有完全无法判断时才使用

2. **sub_concept 必须 ≥ 2 级**，三级可选
3. **note 要写具体**，说明该股票为什么属于这个分类，包含公司主营业务、核心产品、产业链位置等概念相关信息
4. **note 优先使用 reason_info**：如有 `reason_info` 字段，从中提取最近 6 个月内的时效性数据（按优先级：财务数据 > 新产品/技术突破 > 合同/订单 > 政策利好 > 股权变动 > 产能经营），整合为 1-3 句写入 note。无 `reason_info` 时按原有方式（AI 知识 + 搜索）编写
5. **note 禁止使用交易词汇**：不得出现"首板""涨停""跌停""连板""跟风""领涨""爆发""走强""板块调整""概念扩散"等，只描述公司业务和概念归属
6. **note 信息不足时必须联网搜索**：如果 AI 知识库对该公司的了解不足以写出充实的 note（目标 30 字以上），必须联网搜索补充
7. **strength 先全写 5**，不区分
8. **每条记录必须带 `type` 字段**，区分涨跌停
9. **`memberships` 数组至少包含 1 条记录**，不允许为空

## 六、AI 分析优先级规则

`concepts_all.json` 只是历史整理的静态参考数据，**不是绝对正确的标准**。所有股票都需要 LLM 综合分析。

分析每只股票时，按以下因素综合判断：

```
因素 1: 热门概念（板块效应最强）
    该股票是否匹配 热门概念.json 中的热点题材？
    热门概念是基于当天市场行情配置的，优先级最高

因素 2: 概念数量（板块效应）
    同一天涨停股中，同一概念出现的数量越多
    板块效应越强，越应该采用该概念

因素 3: 市场信号（实时标签）
    primary_concept、concepts 和 market_concept 是市场给它的实时标签
    market_concept 来自涨停简图实时分类，可用于补充/确认 real_concepts
    有重要参考价值

因素 4: concepts_all.json（静态参考）
    参考数据中已有的分类，判断是否仍然有效
    注意：可能滞后或过时，不能盲从

因素 5: 业务实质（兜底）
    以上都没有明确指向时，用 AI 知识判断
    仍不确定 → 允许联网搜索
    实在无法判断 → 填 ["未知"]
```

**核心原则：** 综合以上因素，选择最合适的分类。没有哪个因素有绝对的优先级，所有数据都被 LLM 过一遍。

### 判断流程图

判断流程图是对以上因素的可视化总结，展示每只股票的分析决策路径：

```
股票来了
  │
  ├── 热门概念匹配？
  │   ✅ → 优先使用热门概念
  │
  ├── 同概念涨停股数量多？
  │   ✅ → 板块效应强，优先采用
  │
  ├── 市场信号（primary_concept / market_concept）明确？
  │   ✅ → 结合 AI 知识判断
  │
  ├── concepts_all.json 有记录？
  │   ✅ → 判断是否仍然有效
  │
  └── 都没有？
      → AI 知识判断 / 搜索 / 填未知
```

## 七、热门概念.json 说明

```
路径: 题材分析/热门概念.json（可选，不存在时跳过）
```

用于告诉 LLM **当前市场哪些概念在炒作**。LLM 会在全量分析时参考这些概念，判断股票是否与之相关。

**热门概念.json 只有概念名，不强制包含个股。** `items` 是可选字段——大多数时候为空，偶尔你会手动填入几个确认的股票供参考。

### 格式

```json
[
  {
    "concept": "液冷散热",
    "parent_path": ["AI硬件", "散热"],
    "note": "最近谷歌在中国进行液冷的审厂，液冷散热概念持续升温",
    "items": []
  },
  {
    "concept": "低空经济",
    "parent_path": [],
    "note": "政策密集出台，多地发布低空经济发展方案",
    "items": []
  },
  {
    "concept": "玻璃概念",
    "parent_path": ["AI硬件"],
    "note": "AI硬件炒作扩散，上游玻璃基板/玻璃盖板需求增长",
    "items": []
  }
]
```

### 字段说明

| 字段          | 必填 | 说明                                                                                                                                               |
| ------------- | ---- | -------------------------------------------------------------------------------------------------------------------------------------------------- |
| `concept`     | ✅   | 热门概念名称，如"液冷散热"、"低空经济"                                                                                                             |
| `parent_path` | ❌   | 概念在层级中的位置。空数组 `[]` = 一级概念；`["AI硬件"]` = AI硬件下的二级概念；`["AI硬件", "散热"]` = AI硬件/散热下的三级。不填则由 LLM 自行判断。 |
| `note`        | ❌   | 为什么这个概念热，给 LLM 理解背景用                                                                                                                |
| `items`       | ❌   | 可选。你手动确认属于该概念的个股列表。大多数时间为空 `[]`。格式：`[{"symbol": "000823.SZ", "name": "超声电子", "note": "..."}]`                    |

### 概念可以是一/二/三级

- **一级概念**：`"低空经济"`（`parent_path: []`）→ sub_concept: `["低空经济"]`
- **二级概念**：`"玻璃概念"`（`parent_path: ["AI硬件"]`）→ sub_concept: `["AI硬件", "玻璃概念"]`
- **三级概念**：`"液冷散热"`（`parent_path: ["AI硬件", "散热"]`）→ sub_concept: `["AI硬件", "散热", "液冷散热"]`

### 使用方式

LLM 分析每只股票时：

1. 读取热门概念列表
2. 判断该股票的 `concepts`、`primary_concept`、`market_concept`、业务描述是否与热门概念匹配
3. 如果匹配 → 按 `parent_path` + `concept` 拼接成完整层级
4. 不匹配 → 按正常优先级走

### 何时更新

当市场出现**新的持续热点**时，手动在这里加一条。例如：

- 某天涨停数据中有 5 只以上同一概念 → 写进去
- 新闻持续报道某政策/事件驱动 → 写进去
- 某概念明显扩散到上下游 → 写进去

## 八、执行指令

当需要处理某天数据时，按以下顺序操作：

1. **确认日期** — 处理哪一天的数据？
2. **Step 1** — `python py/run_中间.py {date}`（噪音过滤 + concepts_all.json 匹配）
3. **Step 2** — `python py/split_groups.py {date}`（分为5组）→ 将5个分组文件分别交给子LLM分析 → 子LLM产出 `{date}_group_{1..5}_result.json`
4. **Step 3** — **LLM 汇总 + 交叉验证** 5 份 `_result.json`，合并记录、检查重复和一致性、交叉验证，直接产出 `limit_up_分析/{date}.json`（不再产出 `_merged.json`）
5. **Step 4** — （已合并到 Step 3，不再单独执行）
6. **Step 5** — `node py/gen_豆包模板.js {date}`（生成豆包调研模板）
7. **Step 6 — 个股一致性检查** — `python py/validate_stocks.py {date}`（验证原始涨跌停数据与最终输出的个股是否一致）
   - ✅ 通过 → 继续
   - ❌ 有缺失/多余 → **必须排查原因并修复**：
     - 缺失：检查该股票是否在 Step 1 被噪音过滤误杀，或在 Step 3 汇总时被遗漏
     - 多余：检查最终输出中是否混入了非当日涨跌停的股票
     - 修复后重新执行 Step 3 → 再次验证，直到通过
8. **清理中间文件** — 删除 `limit_up_中间/` 中除 `{date}_涨停概念分组_中间.json` 外的所有过程文件（`group_*.json` 等），只保留 `{date}_涨停概念分组_中间.json`；同时删除 `limit_up_分析/` 中残留的 `*_merged.json`（如有）
